# SelfOdds 公开评判标准 v1

SelfOdds 不把所有数字都称为“成功率”。系统明确区分三类结果：

1. **项目机会分（0–100）**：固定权重的决策指数，不是概率。
2. **实验执行成功率（5%–95%）**：Coding Agent 在限定范围内交付并通过验证的预测概率。
3. **7 天趋势概率（5%–95%）**：可自动结算、可计算 Brier Score 的概率预测。

## 项目机会分

| 维度 | 权重 | 评判对象 |
|---|---:|---|
| 需求 | 25% | 明确、反复出现的用户问题；Star 与 README 自述不能单独证明需求 |
| 趋势 | 15% | 近期且持续的活跃或采用，排除一次性新奇效应 |
| 差异化 | 20% | 可服务的空白，而不是逐功能复制 |
| 可构建性 | 20% | 能否在 7 天内完成可证伪的有用实验 |
| 传播性 | 10% | 是否存在可触达用户和自然传播路径 |
| 证据 | 10% | 关键主张能否追溯到元数据、仓库内容或真实结果 |

最终得分由服务端按固定权重计算，模型不能直接决定总分。

### 单项锚点

- `0`：证据反驳该判断，或没有有效信号。
- `25`：只有项目自述或弱代理指标。
- `50`：问题清楚，但关键假设尚未验证。
- `75`：多个独立的已观察信号相互支持。
- `100`：反复出现的真实结果已经充分证明。

### 总分等级

- `A / 80–100`：强实验机会。
- `B / 65–79`：值得测试。
- `C / 50–64`：证据偏弱，只做廉价验证。
- `D / 0–49`：暂不投入。

## 精准需求分析

项目报告必须回答：目标用户、具体问题、当前替代方案、紧迫度、正向证据、反向证据、未知信息和一个可证伪假设。每项重要结论进入证据账本：

- `OBSERVED`：直接出现在仓库元数据、README、仓库结构或用户输入中。
- `INFERRED`：由已观察事实支持的解释。
- `UNKNOWN`：现有证据无法建立该结论。

模型推断不得标记为已观察事实。

## 7 天自动校准

系统保存预测时的 Star 数、活跃时间、概率和到期时间。7 天后自动重新读取 GitHub：

- Star 增长达到 `max(10, round(2 × sqrt(基线 Star)))`；
- 仓库未归档；
- 活跃时间不早于基线快照。

满足全部条件记为 `1`，否则记为 `0`。系统自动计算：

```text
Brier Score = (预测概率 - 实际结果)²
```

机会分不参与概率校准。只有具有明确期限和客观结果的概率预测才进入校准统计。

## 局限

- GitHub 热度只能校准趋势预测，不能证明用户愿意付费。
- README 是项目方自述，应与实际仓库结构和后续行为区分。
- 在积累足够已结算样本前，概率区间比单点概率更值得参考。
- 后续校准器只能使用已结算记录训练，不能使用未验证的模型自评作为标签。
